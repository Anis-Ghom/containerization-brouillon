What the application is for

It is a small internal-style web app for a product catalog and orders. Someone can see products, add new ones, and place orders against existing product ids.

A second column shows orders and a live list of order-related events so you can tell the backend is doing more than static pages.

Slides 1 to 6

How a typical session looks

You open the site through the main entry on port eighty. The page loads products from the first API and orders from the second. Changing a category refreshes the product list only.

Creating a product or an order sends data to the right service. The UI stays simple on purpose so the interesting part is how everything runs in containers and ships through CI.

Slides 1 to 6

What we had to deliver

  • A believable multi-service app, not a single monolith in disguise.
  • Clear container boundaries, health checks, and sensible defaults for data.
  • A pipeline on Gitea that scans for issues and publishes images we can trace back to a commit.

Containers and wiring

Architecture, networks, images, and proof from Docker Desktop.

Slides 7 to 12 · Tom Allory

Traffic flow

The browser talks to Nginx first. Nginx forwards to the Vite dev server and to both APIs on their internal ports.

flowchart LR B[Browser] --> N[Nginx port 80] N --> F[Frontend] N --> A1[API1 Products] N --> A2[API2 Orders] A1 --> P1[(Postgres 1)] A2 --> P2[(Postgres 2)] A2 --> R[(Redis)]

Slides 7 to 12 · Tom Allory

Why four Docker networks

Compose attaches each container only to the networks it needs. The front tier shares one network. Each database sits on a private link with its API. Redis is only on the orders side.

flowchart TB subgraph fn[Frontend network] N[Nginx] F[Frontend] X1[API1] X2[API2] end subgraph n1[API1 and DB1] A1b[API1] D1[(DB1)] end subgraph n2[API2 and DB2] A2b[API2] D2[(DB2)] end subgraph n3[API2 and Redis] A2c[API2] Rd[(Redis)] end

Slides 7 to 12 · Tom Allory

How we build service images

For the Node services we use two stages. The first installs dependencies from the lockfile. The final stage copies only what we need to run, runs as a normal user, and exposes a health check the orchestrator can use.

flowchart LR S1[Stage deps: npm install] --> S2[Final image: app user, healthcheck, start app]

Official images for Nginx, Postgres, and Redis are pinned to specific versions so a rebuild six months later still pulls the same minor line we tested.

Slides 7 to 12 · Tom Allory

Data that survives a restart

Postgres stores data in named volumes so removing a container does not wipe the catalog or orders. The frontend source folder is bind-mounted for local development so you can edit React files without rebuilding the image each time.

flowchart TB H[Your laptop folder] -. bind mount .-> FE[Frontend container] V1[(Volume api1-db-data)] --> DB1[(Postgres 1)] V2[(Volume api2-db-data)] --> DB2[(Postgres 2)]

Product and API behaviour

Main screen

Products on one side, orders and events on the other. Theme can be switched for demos.

Slides 13 to 18 · Ovia Mahalakshmi CHANEMOUGANANDAM

What the frontend asks the APIs

The products list loads from the first service. You can send a new product with name, price, and category. Orders are read from the second service, and you can create an order with a product id and a quantity.

The orders service also exposes a short event feed the UI polls every few seconds so new activity shows up without reloading the whole page.

Slides 13 to 18 · Ovia Mahalakshmi CHANEMOUGANANDAM

Filters and categories

Products can be filtered by category. When you pick a category, only matching rows are requested. If you clear the filter, the full list comes back.

Categories are a fixed set the UI knows about (general, electronics, clothing, food, books, other) so demos stay consistent. GraphQL is available on the products service for alternative clients, but the dashboard uses the REST-style JSON endpoints.

Slides 13 to 18 · Ovia Mahalakshmi CHANEMOUGANANDAM

Behind the UI and how we demo it

Nginx routes traffic to the right service. Redis backs order events on API2. Both APIs ship OpenAPI descriptions if someone wants to try endpoints live during questions.

For a local demo, bring Compose up and use the gateway on port eighty so the jury sees the real proxy path. The APIs stay reachable on their usual host ports if you need a quick check.

Slides 13 to 18 · Ovia Mahalakshmi CHANEMOUGANANDAM

Starting in a safe order

Compose waits until Postgres reports healthy before the APIs start. That avoids connection errors during the first seconds. Health checks on the Node containers give Docker a simple way to see if the process is still answering.

Gitea Actions and Docker Hub

Security scans, builds, tags, and what we push to the registry.

Suggested: Anis Amairi

What runs on every push

The workflow clones the repo, scans configuration files with Trivy, builds each service image locally for scanning, then fails the job if medium or higher issues appear on those images.

Only after that stage succeeds do we push to Docker Hub from the build job. Tagged releases add a human-readable version label that points at the same bits as the commit hash image.

flowchart TD E[Push or PR or version tag] --> SC[Security scan job] SC --> T1[Trivy config on repo] SC --> T2[Build three images for scan] SC --> T3[Trivy image each service] SC --> BP[Build and push job] BP --> H[Docker Hub latest and commit id] BP --> V[Optional extra tag from same digest]

Slides 19 to 24

Scan results for the three microservices

We run the same scanner image the course recommends. These captures are from the pipeline output for frontend, API1, and API2.

Slides 19 to 24

Runs on Gitea Actions

Version labels are created from the digest already pushed for the commit, so we do not rebuild the same Dockerfile twice for a tag.

Slides 19 to 24

Published images and tags

Each microservice has its own repository on Docker Hub. You see latest for the newest main build, a tag that matches the Git commit id, and after a release a short version tag that still points at the same digest as that commit.

flowchart LR I[One digest per build] --> L[latest] I --> S[commit id tag] I --> V[release tag]
